‚ĚĮ it passed, could you please explain why we changed all of those ?             

‚óŹ Sure. The core reason is separation of concerns ‚ÄĒ a principle that each piece
  of code should have one job.

  Page objects have one job: describe the page.
  That means locators and actions ‚ÄĒ things like "here's the title element" and
  "here's how to click checkout". That's it.

  Tests have one job: verify behaviour.
  That means assertions ‚ÄĒ things like "the title should say X" and "there should
   be 6 products".

  When you put expect inside a page object (like the old verifyCartPageTitle),
  the page object is doing two jobs. That causes problems:

  - Reusability breaks ‚ÄĒ what if one test wants to assert the exact text,
  another just wants to check it's visible, another wants to check it's not
  there? If the assertion is hardcoded in the page object, you can't reuse it.
  - Failures are harder to read ‚ÄĒ when a test fails, you want the failure to
  point at the test, not buried inside a page object method.
  - Tests become harder to read ‚ÄĒ verifyCartPageTitle() hides what's actually
  being checked. expect(cartPage.pageTitle).toHaveText('Your Cart') is explicit
  ‚ÄĒ you can read exactly what's being asserted.

  So the rule is simple:

  Page object  ‚Üí  returns locators, performs actions
  Test         ‚Üí  calls page object methods, makes assertions

  That's why verifyCartPageTitle() became get pageTitle() ‚ÄĒ the page object now
  just says "here's the element", and the test decides what to do with it.
